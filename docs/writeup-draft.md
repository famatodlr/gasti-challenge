# Writeup Draft

Construí Gasti como un asistente conversacional de finanzas personales centrado en entender gastos cotidianos en ARS desde chat. La idea no fue hacer una app de budgeting completa ni una demo de dashboards, sino un producto que responda preguntas reales como “qué cambió respecto del mes pasado”, “qué gastos recurrentes tengo” o “a este ritmo cómo cierro el mes”. Para mantener esa experiencia confiable, traté de que los números importantes no dependieran del modelo: los cálculos viven en tools determinísticas y el LLM se usa sobre todo para interpretar intención, decidir cuándo usar cada tool y redactar respuestas claras.

Para meterme en Mastra, primero traté de entender cuáles eran sus primitivas que realmente valían la pena para este problema. La base quedó en un agente principal con tools tipadas por schema, porque eso resolvía muy bien el núcleo conversacional sin inventar un runtime propio. Después sumé workflows solo en casos acotados donde había una orquestación repetible y más estructurada que una conversación abierta: un resumen mensual y un greeting con snapshot corto. También usé memoria conversacional por thread para continuidad entre mensajes, pero separándola de una memoria financiera estructurada propia para no mezclar historial de chat con facts persistentes del usuario demo. Esa separación terminó siendo una de las decisiones más importantes del proyecto.

En el proceso de desarrollo trabajé con IA más como pareja que como autocomplete. Fui iterando a partir de documentos de contexto, decisiones explícitas de alcance y planes de implementación, y después fui validando cada cambio con tests y comandos concretos. Me sirvió bastante usar el repo como fuente de verdad y dejar que la IA ayudara tanto en exploración como en ejecución, pero manteniendo una disciplina fuerte de grounding: revisar cómo estaba realmente implementado el proyecto, qué comandos existían y qué claims se podían sostener con evidencia. Ese mismo criterio lo apliqué al producto: cuando una respuesta dependía de datos o cálculos, preferí empujar esa lógica a dominio y tools antes que dejarla librada al modelo.

Lo que más me costó fue encontrar un punto de equilibrio entre una demo con “memoria” interesante y un comportamiento fácil de explicar. Podía mezclar todo en una sola capa, pero eso me parecía confuso para el usuario y frágil para la demo. También tuve que decidir cuánto scope abrir: había muchas extensiones posibles, pero preferí no agregar features por impresionar y concentrarme en que el núcleo conversacional quedara claro, coherente y defendible. En ese sentido, investigué caminos más ambiciosos como semantic recall o una memoria más libre, pero los dejé afuera porque para este challenge aportaban más complejidad que valor real.

Si siguiera iterando, iría por tres líneas claras: multiusuario real con aislamiento de memoria, una memoria financiera más rica pero igual de controlada, y una capa visual complementaria para tendencias o comparaciones. Me interesaba que esta primera versión mostrara criterio en el uso de Mastra, foco en el valor del producto y un proceso de trabajo con IA que dejara trazabilidad en las decisiones, no solo velocidad para producir código.
